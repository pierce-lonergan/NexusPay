Why NexusPay

Everything a payments team needs — hardened by default.

Sixteen Spring-Modulith modules behind one clean API. Each capability is built money-safe first: idempotent, tenant-scoped, and observable.

Full payment lifecycle

Create, confirm, capture, cancel and retrieve payments with idempotent creates and a server-derived test/live mode.

Double-entry ledger

Every money movement posts balanced journal entries. Queryable accounts and entries — reconciliation built in.

Test mode that is real

sk_test_ keys route to an in-process mock with zero network I/O. A test key can never reach a real processor — enforced by an architecture test.

Scoped, expiring API keys

Per-key scopes (payments:write, refunds:write…) that narrow a role fail-closed, with expiry, last-used tracking and rotate-with-overlap.

Tenant isolation

Zero controllers trust a client tenant header. Every by-id read and mutation resolves through the authenticated principal — no cross-tenant oracle.

Signed, retried webhooks

Canonical versioned events, HMAC-SHA256 signatures, exponential-backoff retries, a dead-letter queue, admin replay and secret rotation.

Refunds & maker-checker

Within-threshold refunds are created directly; large refunds return a pending approval — execute-once and idempotent.

Fraud, sanctions & disputes

A fraud-rules engine + FRM, cross-border & sanctions screening at the payment boundary, and a full dispute/chargeback lifecycle.

Subscription billing

Multiple pricing models, smart-retry dunning, invoicing and proration — with durable test/live routing on every system thread.

How a payment flows

One request in. A signed, reconciled event out.

Your app talks to one API. NexusPay derives the mode from the key, routes test traffic to an in-process mock, posts to the ledger, and delivers a signed webhook you can verify offline.

Get started

From zero to a verified payment, test-safe.

Spin up the lite stack, mint a test key, charge, and verify the webhook — none of it touches a real processor. Full guide in the Integration Guide.

mint-key.sh
# 1 · get an admin token from the dev realm
TOKEN=$(curl -s -X POST \
  http://localhost:8180/realms/nexuspay/protocol/openid-connect/token \
  -d grant_type=password -d client_id=nexuspay-api \
  -d client_secret=nexuspay-api-secret \
  -d username=admin@nexuspay.test -d password=test123 | jq -r .access_token)

# 2 · mint a TEST (live:false) operator key — full key shown ONCE
curl -X POST http://localhost:8090/v1/api-keys \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"name":"my integration","role":"operator","live":false}'
# → response.key starts with sk_test_…  (store as NEXUSPAY_API_KEY)
import { NexusPay } from '@nexus-pay/node';

const client = new NexusPay({
  apiKey: process.env.NEXUSPAY_API_KEY, // sk_test_xxx
  baseUrl: 'http://localhost:8090',
});

const payment = await client.createPayment(
  {
    amount: 600,
    currency: 'USD',
    capture: true,
    metadata: { userId: 'usr_xxx', packId: 'starter' },
  },
  { idempotencyKey: 'order-1234' },
);
// payment.id, payment.status, payment.mode === 'test'
import { constructEvent } from '@nexus-pay/node';

export async function POST(req) {
  const rawBody = await req.text(); // RAW — do not parse first
  let event;
  try {
    event = constructEvent(
      rawBody,
      req.headers,
      process.env.NEXUSPAY_WEBHOOK_SECRET, // whsec_xxx
      { createdToleranceSeconds: 300 },
    );
  } catch {
    return new Response('invalid signature', { status: 401 });
  }
  // event.type === 'payment.succeeded'
  // event.data.metadata === { userId, packId } — credit the right account
}

Security first

Money-safe is the default path, not an option.

Every control fails closed. The platform was hardened through repeated adversarial review — IDOR, PCI, SSRF, money-duplication and authorization-bypass classes were each chased to zero.

Tenant isolation, no oracle

Tenant comes from the authenticated principal — never a client header. By-id reads and mutations 404 on a wrong-tenant id, leaking no existence signal.

Fail-closed scoped keys

Per-key scopes narrow a role with AND-composition; a missing scope is a hard 403. Keys expire, track last use, and rotate with an overlap window that never extends a key's life.

PCI-aware card handling

Card data is encrypted at rest with AES-256-GCM behind a master key — never base64, never plaintext. Raw PAN never persists.

SSRF-pinned webhooks

Outbound delivery resolves and pins the IP, refuses redirects and private ranges, and validates at registration and delivery. HMAC-SHA256 signed; verifiable offline.

Idempotent money moves

Capture, void and refund take an idempotency key; webhooks dedupe before commit; the ledger never double-posts and payouts never double-pay.

Maker-checker on big refunds

Refunds above a threshold require approval and execute exactly once — across every path, including the off-request settlement route.

Genuinely built for

  • A legible, security-first reference architecture for a hardened multi-tenant payments backend on HyperSwitch.
  • Teams for whom test-mode developer experience is a priority — arguably ahead of its OSS peers.
  • Researchers, students, nonprofits, or anyone evaluating the patterns under the noncommercial license.
  • Studying zero-trust tenancy, HMAC+replay webhooks, idempotency, maker-checker and outbox done right.

Not the right fit if

  • You're an early-stage or low-volume merchant — buy a hosted processor (Stripe/Square) and ship in hours.
  • You're a commercial team that would otherwise reach for HyperSwitch, Kill Bill, Lago or Formance — those are permissively licensed and funded.
  • You need external attestation (SOC 2 / PCI Level 1 / pentest) or a support SLA today.
  • You expect it to move money — it orchestrates and records; it does not settle funds.

Contribute

Built in the open, reviewed adversarially.

Contributions are welcome. Every change ships through CI — and money-touching changes get an adversarial security review. Start with the contributing guide.

01

Fork & branch

Fork the repo and branch from main. Run the lite stack locally (no real money moves in test mode).

02

Build with tests

Add tests with your change — fail-closed first. The ratchet enforces a rising test floor and coverage.

03

Open a PR

CI runs build, tests, secret/dependency/SAST scans and a red-team simulation. Keep the diff focused.

04

Pass the gates

A green PR + (for money paths) an adversarial review gets merged. Decisions are recorded as ADRs.

✓ build & test ✓ coverage + test-count ratchet ✓ secret scan ✓ dependency scan ✓ SAST (semgrep) ✓ red-team simulation
